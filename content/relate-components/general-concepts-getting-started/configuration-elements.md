<div class="pageHeader">Configuration Elements</div>
<div style="margin-bottom: 6px" class="pageRule"><hr></div>




<p>A Relate custom application is a collection of elements or components.&nbsp; These elements are created, configured and connected to one another via the Relate site.&nbsp; Some of these elements can be connected to webpages on the other websites throughout the system.&nbsp; These external connections are&nbsp;configured on the individual&nbsp;websites.&nbsp; Below is a list of Relate element types.&nbsp; The next article in this series describes potential external connections to&nbsp;Relate.&nbsp; Listed with each is a brief description&nbsp;of the function and capabilities of that element.</p>
<p><strong>Important concept:</strong> &nbsp;Most types of Relate configuration elements have both a name and a label. &nbsp;The name is required, but the label is optional. &nbsp;The name is what the element is called for the Relate configuration person and is shown on all configuration screens. &nbsp;The label is what the element is called when the end user sees it. &nbsp;The label is shown while entering data and viewing reports. &nbsp;If no label is given, then the name is seen both on configuration screens and on end-user screens. &nbsp;During advanced scripting logic some items need to be looked up by name. &nbsp;The name can be set and never changed since only the configuration staff will ever see it. &nbsp;This allows for reliable look-up by name. &nbsp;The label, on the other hand, can change based on the whims of whoever is in charge and their preferences at the moment without breaking any scripting logic.</p>
<p><span><a href="/relate-components/other-elements-and-functions/record-types-and-categories">Record Types and Categories</a></span><br>These elements were discussed in the previous article.&nbsp; They allow forms and form entries to be associated with Relate records.&nbsp; Each record type has exactly one private form that is unique to that record type and contains, at minimum, a field representing the&nbsp;name of the record.&nbsp; All other forms in the system can be connected&nbsp;to as many record types and categories&nbsp;as the designer wishes.</p>
<p><a href="/relate-components/other-elements-and-functions/bundles">Bundles</a><br>Bundles allow all of the other types of Relate elements to be grouped together into projects or bundles. &nbsp;This provides numerous benefits for large scale Relate configurations such as: &nbsp;1. Gathering components which may be spread across many areas of Relate into a common place where their relationships to each other can be recorded and described. 2. Providing version notes so changes over time can be described and reviewed. &nbsp;3. Finding and highlighting changes that have occurred since a specific timestamp. &nbsp;4. &nbsp;Locking elements from unauthorized modification and requiring note taking when authorized users make changes.</p>
<p><a href="/relate-components/relate-structure/fields"><span>Forms and Fields</span><br></a>As discussed a form is&nbsp;a collection of fields.&nbsp; Each field has a data type such as text, number or date.&nbsp; Each field also has a name that identifies the&nbsp;bits of data that will be&nbsp;entered using the form.&nbsp; The fields are the core of any Relate application.&nbsp; Because of this, each field has numerous&nbsp;settings which allow its appearance and behavior to be adapted to a wide range of uses.&nbsp; &nbsp;The system will automatically generate a default view of the form, based on the field settings,&nbsp;which allows the form to be filled out and an entry created.&nbsp;&nbsp; Remember, however,&nbsp;that a form never stands alone.&nbsp; Until&nbsp;the form&nbsp;is associated with one or more record types and/or record categories it is entirely useless.&nbsp; Below is a quick list of the types of data that can be configured and entered via a field element.</p>
<ul>
<li><span><a href="/relate-components/relate-structure/fields/text-fields">Text</a></span> - Text with a maximum length of 4000 characters.&nbsp; The text may be formatted in various ways such as a phone number, a U.S. state abbreviation, an email address, etc.&nbsp; Text fields can also present a list of choices to select from, and the chosen text is stored in the field. &nbsp;Text fields can also be used to select-by-searching from some very large pre-defined lists such as the list of all medications licensed by the FDA.</li>
<li><span><a href="/relate-components/relate-structure/fields/memo-fields">Memo</a></span> - Multiple lines of text.&nbsp; This can also be used to store a chunk of HTML or a list of email addresses (but nobody ever does that...I'm not sure why).</li>
<li><span><a href="/relate-components/relate-structure/fields/date-time-fields">Date/Time</a></span> - Stores a date, a time or both.&nbsp; Dates and times stored separately are always displayed as entered, but fields storing both date and time are converted to the current user's preferred timezone whenever they are displayed or edited.</li>
<li><span><a href="/relate-components/relate-structure/fields/number-fields">Number</a></span> - Stores a number in the mathematical sense (never a phone number or social security number).&nbsp; Integers, real numbers or currency values can be stored.&nbsp; Currency values are just a real number with two decimal places.&nbsp; They don't do currency conversions or track what currency is being stored.</li>
<li><span><a href="/relate-components/relate-structure/fields/boolean-fields">Boolean</a></span> - Boolean is the mathematical term for things with only two possible values.&nbsp;&nbsp;Boolean fields&nbsp;store true/false, yes/no, paper/plastic or any number of other simple one-or-the-other choices.</li>
<li><span><a href="/relate-components/relate-structure/fields/single-select-fields">Single Select</a> </span>-&nbsp;This type of field displays a list of choices and allows exactly one to be selected.&nbsp; The list of possible values is configured via an option list.&nbsp; The field stores a reference to the option in the option list, not the actual text of the option.&nbsp; Because of this, changing the&nbsp;label of an option changes how that option will be displayed in every form entry where it has been selected throughout the system.</li>
<li><span><a href="/relate-components/relate-structure/fields/multi-select-fields">Multi-Select</a></span> - Nearly identical to a single select field except multiple options can be selected.&nbsp; I bet you figured that out before you finished the sentence.</li>
<li><span><a href="/relate-components/relate-structure/fields/signature-fields">Signature</a> </span>- Stores&nbsp;a timestamp and a reference to the user's account when the field is 'signed' usually by checking a check-box.&nbsp; After signing the field becomes read-only and shows the stored timestamp and the name of the user who signed.&nbsp; Signature fields are unique in that they cannot be completed in any way other that the end user actually checking the box.&nbsp; All other field types can be manipulated via numerous methods, but the only modifying action allowed on a signature field is to unsign it.&nbsp; Even&nbsp;unsigning can only be performed by&nbsp;extremely privileged users: an&nbsp;organization administrator (by editing the entry and unchecking the box)&nbsp;or a relate administrator (via formula).</li>
<li><span><a href="/relate-components/relate-structure/fields/document-fields">Document</a></span> - Stores a link to a document or file.&nbsp; This type of field also facilitates uploading new documents and creating and manipulating documents via formula.&nbsp; If the file is an image&nbsp;then&nbsp;the field can generate a thumbnail.&nbsp; The documents thus linked are stored with and connected to the Relate record in a special document library.&nbsp; Each Relate record has its own library of documents and files which belong to that record and cannot be&nbsp;connected to a document field in another record.</li>
<li><span><a href="/relate-components/relate-structure/fields/relationship-fields">Relationship</a></span> - In appearance this type of field usually looks like a single select or multi-select field:&nbsp; It presents a list of options to be selected.&nbsp; However, the options instead of being from a semi-static option list are actually other form entries stored either within the current record or in other records in the system.&nbsp; Selecting an option makes a connection or relationship between the current entry and the entry selected.&nbsp; This relationship can be seen and changed from the entry on either end of the relationship.&nbsp; These fields are difficult to configure correctly, but can be used&nbsp;in very&nbsp;powerful ways.</li>
<li><span><a href="/relate-components/relate-structure/fields/spacer-fields">Spacer</a></span> - Not a true field in the strict sense.&nbsp; This is merely a place holder that directs the system building the automatic layout of the form to add some vertical space between two fields.</li>
<li><span><a href="/relate-components/relate-structure/fields/section-header-fields">Section Header</a></span> - Also not a true field, this type directs the automatic layout system&nbsp;to include a section header to visually separate the fields into logical groupings.</li>
<li><a href="/relate-components/relate-structure/fields/merge-report-fields">Merge-Report Field</a> - Again, not a true field. &nbsp;This type of 'field' allows a custom layout to be inserted in the middle of an automatic layout. &nbsp;Often the automatic layout is exactly what you need...well almost exactly what you need. &nbsp;In case of 'almost' a merge-report field fills in the gap by allowing a portion of the layout to be custom while retaining the remaining automatic layout. &nbsp;The automatic layout is able to do many complex functions which are difficult or impossible to reproduce in a custom layout, but by creating a hybrid of the two concepts you can get the best of both worlds.</li>
<li><a href="/relate-components/relate-structure/fields/biometric-field">Biometric</a> - A biometric field recieves input from specialized biometric hardware input devices. &nbsp;Not all biometric devices are supported since each requires custom programming. &nbsp;Currently, the biometric field supports certain fingerprint scanners and signature pads.</li>
</ul>
<p>Certain types of fields can be used for <span>grouping</span>. &nbsp;The grouping features are different for different types of fields, but they share a common thread: &nbsp;groups of fields may become sub-fields to a master field. &nbsp;The master field determines if and when the sub-fields will be visible. &nbsp;This is a very powerful concept for creating dynamic and friendly forms.&nbsp; Grouping fields is simple:&nbsp; When grouping is turned on for the master field, it splits and becomes two (or more) elements in the field listing.&nbsp; To put other fields in the group you must reorder the fields so the grouped fields are between the parts of the master field.</p>
<p><a href="/relate-components/relate-structure/option-lists-and-groups"><span>Option Lists</span><br></a>As the name implies, this is a list of options.&nbsp; Each option has a label, an optional export value, display options and identifiers for use in RelateScript.&nbsp; Option lists are used by text fields, single select fields and multi-select fields to compose a list of options to be selected from.</p>
<p><span><a href="/relate-components/relate-structure/wizards">Wizards</a></span><br>A wizard is used to direct the user through a series of forms to be filled out together.&nbsp; Wizards are the normal way to create a new record, although the system auto-generates some basic wizards for this task.&nbsp; Wizards can be used to add a new category to an existing record such as when converting a sales lead to an active client.&nbsp; Lastly, a special type of wizard, a user wizard, is used to allow a user to enter data into their own account.&nbsp; User wizards can be used as a sign-up mechanism on sites which allow users to create accounts at will.&nbsp; Normal or 'general' wizards can be shown in&nbsp;record navigation and thus be used to edit the record and/or add a record category.&nbsp; Technically users wizards can be shown in record navigation, but this is seldom a good idea.</p>
<p><a href="/relate-components/relate-structure/merge-reports"><span>Merge Reports</span><br></a>Merge reports are so called because they function much like a mail merge in a word processor.&nbsp;&nbsp; A merge report consists of a block of HTML code with special merge tags inserted throughout where Relate data is to be inserted.&nbsp; Merge tags can insert data as either read-only or editable values.&nbsp; Merge reports are often used to create alternate layouts of a form when the auto-generated layout does not meet a particular need.&nbsp; They can also pull in data from multiple forms onto one screen allowing data from all over a record to be edited or displayed on a single page.&nbsp; Merge tags can also pull in another merge report allowing re-use of common building blocks. &nbsp;These building blocks are often combined to create complex layouts from simpler pieces.&nbsp; Merge tags can insert an entire multi-entry report (see next paragraph).&nbsp; Lastly, merge reports can contain formulas which read data from throughout the system and do complex calculations or sophisticated user interactions.&nbsp; Merge reports can be shown in&nbsp;record navigation, be selected as an alternate layout of a form entry being edited or inserted as a merge-report field.&nbsp; They can be used in numerous other ways which will be touched on later in this document.</p>
<p><a href="/relate-components/relate-structure/multi-entry-reports"><span>Multi-Entry Reports</span><br></a>Composes a report from the entries of one multi-entry form contained within exactly one Relate record..&nbsp; The report is a table with rows representing entries of the form and columns representing fields on the form.&nbsp; A partial list of entries can be composed using search criteria such as showing entries from the past 30 days.&nbsp; The entries can be sorted by any sortable column (some types of data aren't sortable).&nbsp; Also numeric columns may be totaled, sometimes with sub-totals by grouping entries with a common value in&nbsp;a chosen field.&nbsp;&nbsp;A multi-entry report may be configured to display a merge report in&nbsp;a column, repeated for each entry&nbsp;being displayed.&nbsp; In this way multi-entry reports can be used to show much more complex layouts than simple rows and columns.&nbsp;&nbsp; Also&nbsp;the merge-report-within-a-multi-entry-report&nbsp;technique can be used to edit many entries of the form all at once by choosing a merge report with editable values embedded in it.&nbsp;&nbsp;Multi-entry reports are often used to present a shortened list of entries from which to select an entry for edit when the total list contains too much historical information to be practical in everyday usage.</p>
<p><span><a href="/relate-components/relate-structure/formulas">Formulas</a></span><br>Formulas use a simple proprietary computer scripting language call Relate Script.&nbsp; Relate script formulas can be executed on a&nbsp;schedule or be triggered by events in the system such as editing or deleting&nbsp;an entry of a particular form.&nbsp; Formulas can be used to do complex calculations, send notifications via email and other messaging systems, check newly entered data for compliance with complex rules, and many other things.&nbsp; A discussion of the capabilities of formulas is beyond the scope of this document.&nbsp; Read the Relate Script documentation for more complete information.</p>
<p><a href="/relate-components/relate-structure/end-points"><span>End Points</span><br></a>End points are advanced formulas that can respond to a raw HTTP request and generate an HTTP response.&nbsp;&nbsp; They are used to create complex user interactions and communicate with other systems.&nbsp;&nbsp;Only BlueStep engineers and other specially trained individuals&nbsp;can create and edit end points because their extreme power also makes them extremely dangerous.</p>
<p><span><a href="/relate-components/relate-structure/folders">Folders</a></span><br>Folders allow relate elements to be created and navigated via a hierarchical folder structure.&nbsp; A large Relate application may contain several thousand elements.&nbsp; If not carefully organized and grouped into folders and sub-folders this would be totally overwhelming and impossible to deal with.&nbsp; Folders are always used to organize Relate elements&nbsp;for configuration within the Relate website.&nbsp; They may also be shown on the Relate navigation of a record to allow the forms and other visible elements to be grouped and collapsed&nbsp;as the user navigates the record.</p>
<p><span><a href="/relate-components/other-elements-and-functions/queries-reports">Queries and Reports</a><br></span>Queries and reports are much like multi-entry reports (or rather multi-entry reports are like queries and reports).&nbsp;&nbsp; They display data in a table layout with rows and columns.&nbsp; The rows of a query or report, instead of representing an entry of a form, represent a record.&nbsp; And the columns show fields or merge-reports, but they may be from any form or merge-report associated with the records being displayed.&nbsp; Reports have nearly all of the same features as multi-entry reports in terms of searching and display capabilities.&nbsp; The only missing feature is they cannot be used to display editable fields.&nbsp; This is because Relate currently only allows one record to be edited at a time.&nbsp; Reports have one feature not available in multi-entry reports:&nbsp; When a row of the report is selected, the report can direct the user to almost any form&nbsp;or merge-report associated with the record.&nbsp; After viewing the report or editing the form entry, the system can automatically return the user to the report to select another record.&nbsp; Queries are nearly identical to reports.&nbsp; They&nbsp;have the additional feature of being able to be searched by the end user while viewing the query results.&nbsp;&nbsp;Because of this, queries are most commonly used to find a record to work with.&nbsp; Query's results are also paginated when they are displaying large numbers of records and they do not allow grouping or totalling.&nbsp; Queries and Reports can also be used to export data from the system in CSV format.</p>
<p><span style="color: #426c95; font-size: 12pt; font-weight: bold;">Exercise</span></p>
<p>Unless you are in a test orgnaization, be very careful doing this exercise.&nbsp; Any change you make will become effective immediately and may confuse your users or even break&nbsp;existing configuration.</p>
<p>Try to create at least one of each type of element that you can (most users can't create end points, for instance).&nbsp; Note that record types, folders, option lists, forms and most field types can be created basically stand-alone,&nbsp;without referencing other pieces of configuration.&nbsp; However,&nbsp;many element types&nbsp;require that other elements exist.&nbsp;&nbsp;In most cases these elements require&nbsp;that&nbsp;forms and fields exist and that the forms be connected to record types and/or categories.&nbsp; A few types of elments (relationship fields and merge report fields in particular) are quite difficult to create correctly&nbsp;because they require many carefully configured elements to pre-exist.&nbsp; Of course, creating a formula will require some knowlege of the BlueStep scripting language.&nbsp; Full documentation and reference material&nbsp;for&nbsp;Relate Script&nbsp;can be found at <a href="https://bluestepplatformsupport.bluestep.net/shared/custompage/custompage.jsp?_event=view&amp;_id=445506_U129801__3121">clientcare.bluestep.net</a> (this site).&nbsp; As you create various elements and connect them to each other, observe how your changes effect the applicable records which use these elements.</p>
